# Switch

Switches provide intuitive binary control with immediate visual feedback. The ELEVATE Switch component helps designers and developers create consistent, accessible toggle interfaces that clearly communicate state changes and system responses.

## Overview

### When to Use Switches
- **Settings and preferences**: Enable/disable features and system options
- **Instant state changes**: Controls that take effect immediately when toggled
- **Binary choices**: On/off, enabled/disabled, public/private selections
- **Feature toggles**: Activate or deactivate functionality within interfaces
- **Notification controls**: Manage alert preferences and communication settings

### When NOT to Use Switches
- **Form submissions**: Use checkboxes for options that require form submission
- **Multiple related options**: Use checkbox groups for selecting multiple items
- **Temporary states**: Use buttons for actions that trigger processes
- **Destructive actions**: Use buttons with confirmation for irreversible changes

## Design Principles

### Immediate Feedback Principle
**State changes should be instantly visible and clear.** Users need immediate confirmation that their toggle action has been registered and applied without requiring form submission or page refresh.

### Predictable Behavior Principle  
**Switch appearance should clearly indicate current state and expected behavior.** Users should be able to predict what will happen when they toggle a switch based on its visual design and context.

### Semantic Clarity Principle
**Switch states should have obvious meanings that align with user expectations.** On/off, enabled/disabled meanings should be culturally appropriate and contextually clear.

### Reversibility Principle
**Users should understand that switch actions are immediately reversible.** Switches are for temporary, easily changed states rather than permanent or destructive actions.

## Best Practices

### Providing Clear Labels

**Why this matters:** Ambiguous switch labels leave users uncertain about what they're controlling. Clear, descriptive labels help users understand the impact of their toggle actions.

:::tip Do: Use specific, actionable labels
Write labels that clearly describe what will happen when the switch is toggled.

```html
<elvt-switch>
  <label slot="label">Email notifications</label>
  <span slot="help-text">Receive alerts for important updates</span>
</elvt-switch>

<elvt-switch checked>
  <label slot="label">Auto-save documents</label>
  <span slot="help-text">Automatically save changes every 30 seconds</span>
</elvt-switch>
```
:::

:::danger Don't: Use vague or confusing labels
Avoid generic labels that don't explain what the switch controls.

```html
<!-- Avoid this -->
<elvt-switch>
  <label slot="label">Enable feature</label>
</elvt-switch>

<elvt-switch>
  <label slot="label">Toggle setting</label>
</elvt-switch>
```
:::

### Indicating Current State

**Why this matters:** Users need to understand whether a switch is currently on or off before they interact with it. Poor state indication leads to confusion and unintended actions.

:::tip Do: Make switch states visually distinct
Ensure on and off states are clearly different and follow user expectations.

```html
<elvt-switch checked>
  <label slot="label">Dark mode (Currently on)</label>
</elvt-switch>

<elvt-switch>
  <label slot="label">Notifications (Currently off)</label>
</elvt-switch>
```
:::

:::danger Don't: Rely only on subtle visual differences
Avoid state indicators that are hard to distinguish or unclear.

```html
<!-- Avoid unclear state indication -->
<elvt-switch style="opacity: 0.7;">
  <label slot="label">Unknown state</label>
</elvt-switch>
```
:::

### Choosing Appropriate Context

**Why this matters:** Switches work best for immediate, reversible state changes. Using switches for actions that require confirmation or have delayed effects creates misleading user expectations.

:::tip Do: Use switches for immediate, reversible actions
Reserve switches for settings that take effect immediately and can be easily changed.

```html
<elvt-switch>
  <label slot="label">Show profile to public</label>
  <span slot="help-text">Changes visibility immediately</span>
</elvt-switch>
```
:::

:::danger Don't: Use switches for destructive or complex actions
Avoid switches for actions that require confirmation or have permanent effects.

```html
<!-- Avoid this - use button with confirmation -->
<elvt-switch>
  <label slot="label">Delete all data</label>
</elvt-switch>
```
:::

### Basic Usage

```html
<!-- Basic switch -->
<elvt-switch>
  <label slot="label">Enable notifications</label>
</elvt-switch>

<!-- Pre-checked switch -->
<elvt-switch checked>
  <label slot="label">Dark mode</label>
</elvt-switch>

<!-- Disabled switch -->
<elvt-switch disabled>
  <label slot="label">Maintenance mode</label>
</elvt-switch>

<!-- Switch with description -->
<elvt-switch>
  <label slot="label">Auto-save</label>
  <span slot="help-text">Automatically save changes every 30 seconds</span>
</elvt-switch>

<!-- Compact switch -->
<elvt-switch size="small">
  <label slot="label">Compact view</label>
</elvt-switch>

<!-- Switch group -->
<fieldset>
  <legend>Privacy Settings</legend>
  <elvt-switch checked>
    <label slot="label">Profile visibility</label>
  </elvt-switch>
  <elvt-switch>
    <label slot="label">Show online status</label>
  </elvt-switch>
  <elvt-switch checked>
    <label slot="label">Allow direct messages</label>
  </elvt-switch>
</fieldset>
```

### Key Properties

| Property | Type | Description |
|----------|------|-------------|
| `checked` | Boolean | Whether switch is in on state |
| `disabled` | Boolean | Whether switch can be toggled |  
| `size` | `small`, `medium` (default), `large` | Physical size of switch |
| `name` | String | Form field name for data submission |
| `value` | String | Form field value when switch is on |

**Slots:** Use `slot="label"` for the switch label and `slot="help-text"` for additional descriptive text.

*For complete API reference including all properties, events, and methods, see the [Storybook documentation](https://elevate-core-ui.inform-cloud.io/?path=/docs/components-switch--docs).*

## Accessibility

### Essential Accessibility Practices

**Keyboard interaction:**
- Space key toggles switch state
- Tab navigation moves between switches
- Clear focus indicators show current focus
- Enter key can also activate switches

**Screen reader support:**
- Use proper ARIA roles (switch or checkbox)
- Announce current state (on/off, checked/unchecked)
- Associate labels with switch controls
- Provide descriptions for complex switches

**Visual accessibility:**
- Ensure sufficient color contrast for all states
- Don't rely solely on color to indicate state
- Provide clear visual differences between on/off states
- Include text or icons alongside color changes

**State communication:**
- Announce state changes to assistive technology
- Provide immediate feedback for user actions
- Use consistent language for similar switches
- Include context about what the switch controls
