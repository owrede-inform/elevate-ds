Button Groups combine multiple related buttons into a cohesive unit with consistent spacing, alignment, and visual treatment. The ELEVATE Button Group component helps designers and developers organize actions logically, create clear visual hierarchy, and reduce interface complexity while maintaining accessibility and usability standards.

## Overview

### When to Use Button Groups
- **Related actions**: Buttons that perform similar or complementary functions
- **Action sets**: Save/Cancel, Previous/Next, Accept/Decline pairs
- **Toolbars**: Sets of tools or commands in applications
- **Bulk actions**: Multiple operations that can be performed on selected content
- **Workflow steps**: Sequential actions in a multi-step process
- **Form actions**: Multiple submission or navigation options

### When NOT to Use Button Groups
- **Unrelated actions**: Buttons that serve completely different purposes
- **Primary navigation**: Use appropriate navigation components instead
- **Single actions**: Individual buttons don't need grouping treatment
- **Complex forms**: Use proper form layout patterns with field grouping
- **Selection options**: Use Radio buttons, Checkboxes, or Select components

## Design Principles

Understanding these core principles will help you create button groups that enhance rather than complicate user interactions and decision-making.

### Visual Hierarchy Principle
**Button groups should have clear primary and secondary actions.** Users need to quickly identify the most important action within a group. Equal visual weight on all buttons creates decision paralysis and slows task completion.

### Cognitive Grouping Principle  
**Related actions should be visually and functionally grouped together.** Logical organization reduces cognitive load and helps users predict what actions are available. Random grouping confuses users and breaks workflow expectations.

### Spatial Relationship Principle
**Physical proximity should reflect functional relationship.** Buttons that work together should be close together, while destructive or opposing actions should have visual separation to prevent accidental activation.

### Contextual Relevance Principle
**All buttons in a group should be relevant to the current context.** Including unrelated actions in a group dilutes the purpose and forces users to evaluate irrelevant options, increasing interaction time and error rates.

## Best Practices

### Establishing Clear Action Hierarchy

**Why this matters:** When multiple buttons in a group have equal visual weight, users can't quickly identify the primary action. This creates hesitation and increases the likelihood of selecting the wrong action, especially in high-stakes situations.

:::tip Do
Use clear visual hierarchy with one primary action per group

Visual example showing primary button (solid blue), secondary button (outlined), and tertiary button (subtle gray)
:::

:::danger Don't
Use multiple primary buttons that compete for attention

Visual example showing three buttons all with the same solid blue styling
:::

### Organizing Logically Related Actions

**Why this matters:** Grouping unrelated actions together confuses users about the purpose of each group and forces them to evaluate irrelevant options. Logical grouping accelerates task completion by presenting contextually relevant choices.

:::tip Do
Group related actions that work together in the same workflow

Example: Copy, Cut, Paste (editing actions) / Undo, Redo (history actions)
:::

:::danger Don't
Mix unrelated actions that serve different purposes

Example: Copy, Settings, Help, Paste (mixing editing with navigation actions)
:::

### Key Properties

The most commonly used properties for design and development:

| Property | Options | Description |
|----------|---------|-------------|
| `orientation` | `horizontal` (default), `vertical` | Layout direction of the button group |
| `variant` | `default`, `segmented`, `toolbar` | Visual treatment and connection style |
| `spacing` | `small`, `medium` (default), `large` | Space between buttons in the group |
| `alignment` | `start` (default), `center`, `end`, `stretch` | Button alignment within the group |
| `wrap` | `true`, `false` (default) | Whether buttons can wrap to new lines |
| `full-width` | `true`, `false` (default) | Whether group takes full container width |

*For complete API reference including all properties, events, and methods, see the [Storybook documentation](https://elevate-core-ui.inform-cloud.io/?path=/docs/components-buttons-button-group--docs).*

## Accessibility

### Essential Accessibility Practices

**Group semantics when needed:**
```html
<!-- Good: Semantic grouping for related controls -->
<div role="group" aria-label="Text formatting">
  <elvt-button-group>
    <elvt-button aria-label="Bold">
      <elvt-icon icon="mdi:format-bold"></elvt-icon>
    </elvt-button>
    <elvt-button aria-label="Italic">
      <elvt-icon icon="mdi:format-italic"></elvt-icon>
    </elvt-button>
  </elvt-button-group>
</div>

<!-- Bad: Missing context for screen readers -->
<elvt-button-group>
  <elvt-button><elvt-icon icon="mdi:format-bold"></elvt-icon></elvt-button>
  <elvt-button><elvt-icon icon="mdi:format-italic"></elvt-icon></elvt-button>
</elvt-button-group>
```

**Keyboard navigation works automatically:**
- Tab moves focus between buttons in the group
- Arrow keys navigate between buttons in segmented groups
- Enter and Space activate the focused button
- Home and End move to first/last button in arrow key navigation

**Icon-only buttons need labels:**
- ✅ Always provide `aria-label` or `title` for icon-only buttons
- ✅ Use descriptive labels that explain the action
- ❌ Don't rely on icons alone for meaning

**Ensure sufficient contrast:**
- All button text and borders must meet 4.5:1 contrast ratio minimum
- Focus indicators must be clearly visible
- Disabled states should still be readable

**Group identification:**
- Use `role="group"` with `aria-label` for related button sets
- Segmented groups use `role="radiogroup"` for single selection
- Selected states use `aria-pressed` or `aria-selected`
