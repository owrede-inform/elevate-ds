# Input

Input components are essential form elements that allow users to enter text, numbers, and other data. The ELEVATE Input component provides consistent styling, validation states, and accessibility features across all INFORM applications.

## Overview

### When to Use Inputs
- **Data entry**: Text, numbers, emails, passwords, and other user input
- **Search functionality**: Search boxes and filtering interfaces
- **Forms**: Registration, contact forms, settings, and configuration
- **Content creation**: Comments, posts, descriptions, and text content

### When NOT to Use Inputs
- **Binary choices**: Use Switch or Checkbox for yes/no selections
- **Multiple selections**: Use Select, Dropdown, or Checkbox group for multiple options
- **Large text**: Use Textarea component for multi-line text input
- **Dates**: Use DatePicker for date selection with calendar interface

## Design Principles

### Clarity Principle
**Input purpose and requirements must be immediately clear.** Users should understand what information is expected and how to format it correctly without guessing or trial-and-error.

### Feedback Principle
**Provide immediate, helpful feedback for user actions.** Validation messages should appear when relevant and offer specific guidance for correction, not just indicate that something is wrong.

### Accessibility Principle
**All users must be able to perceive, understand, and operate inputs.** This includes proper labeling, keyboard navigation, screen reader support, and sufficient color contrast.

### Consistency Principle
**Similar input types should behave similarly across the application.** Consistent patterns reduce cognitive load and help users transfer learning between different parts of the interface.

## Best Practices

### Writing Effective Labels and Help Text

**Why this matters:** Unclear labels and missing help text force users to guess what information is expected, leading to errors, frustration, and form abandonment.

:::tip Do: Use clear, descriptive labels with helpful guidance
Provide specific labels and actionable help text that guide users toward success.

```html
<elvt-input 
  type="email" 
  label="Work Email Address" 
  placeholder="name@company.com"
  help-text="We'll use this to send you important account updates"
  required>
</elvt-input>
```
:::

:::danger Don't: Use vague labels without context
Avoid generic labels that don't specify requirements or expected format.

```html
<!-- Avoid this -->
<elvt-input 
  type="text" 
  label="Email" 
  placeholder="Email">
</elvt-input>
```
:::

### Providing Clear Validation Feedback

**Why this matters:** Poor validation messages frustrate users and prevent successful form completion. Good validation helps users understand exactly what they need to fix.

:::tip Do: Provide specific, actionable error messages
Tell users exactly what's wrong and how to fix it.

```html
<elvt-input 
  type="password" 
  label="Password" 
  invalid
  help-text="Password must contain at least 8 characters, including 1 number and 1 symbol">
</elvt-input>
```
:::

:::danger Don't: Use generic or unhelpful error messages
Avoid vague messages that don't help users understand how to fix the problem.

```html
<!-- Avoid this -->
<elvt-input 
  type="password" 
  label="Password" 
  invalid
  help-text="Invalid password">
</elvt-input>
```
:::

### Using Appropriate Input Types

**Why this matters:** Correct input types provide better mobile keyboards, built-in validation, and improved accessibility. Wrong types create friction and confusion.

:::tip Do: Use specific input types for different data
Choose the most appropriate input type for the expected data format.

```html
<elvt-input type="email" label="Email Address"></elvt-input>
<elvt-input type="tel" label="Phone Number"></elvt-input>
<elvt-input type="url" label="Website URL"></elvt-input>
<elvt-input type="number" label="Age"></elvt-input>
```
:::

:::danger Don't: Use generic text inputs for structured data
Don't use text inputs when more specific types would provide better experience.

```html
<!-- Avoid this -->
<elvt-input type="text" label="Email Address"></elvt-input>
<elvt-input type="text" label="Phone Number"></elvt-input>
<elvt-input type="text" label="Website URL"></elvt-input>
```
:::

### Common Usage Patterns

```html
<!-- Basic text input -->
<elvt-input 
  type="text" 
  label="Full Name" 
  placeholder="Enter your full name"
  required>
</elvt-input>

<!-- Email input with validation -->
<elvt-input 
  type="email" 
  label="Email Address" 
  placeholder="user@example.com"
  help-text="We'll never share your email address"
  required>
</elvt-input>

<!-- Password input -->
<elvt-input 
  type="password" 
  label="Password" 
  help-text="Must be at least 8 characters"
  required>
</elvt-input>

<!-- Input with validation state -->
<elvt-input 
  type="text" 
  label="Username" 
  value="user123"
  valid
  help-text="Username is available">
</elvt-input>

<!-- Disabled input -->
<elvt-input 
  type="text" 
  label="Account ID" 
  value="ACC-123456"
  disabled>
</elvt-input>
```

### Key Properties

| Property | Type | Description |
|----------|------|-------------|
| `type` | `text`, `email`, `password`, `tel`, `url`, `number`, `search` | Input type for appropriate keyboard and validation |
| `label` | String | Accessible label text (required) |
| `placeholder` | String | Placeholder text shown when empty |
| `help-text` | String | Additional guidance or validation message |
| `value` | String | Current input value |
| `size` | `small`, `medium` (default), `large` | Visual size of the input |
| `required` | Boolean | Whether the field is required |
| `disabled` | Boolean | Whether the input is disabled |
| `readonly` | Boolean | Whether the input is read-only |
| `valid` | Boolean | Shows positive validation state |
| `invalid` | Boolean | Shows error validation state |
| `warning` | Boolean | Shows warning validation state |
| `maxlength` | Number | Maximum number of characters allowed |
| `minlength` | Number | Minimum number of characters required |

*For complete API reference including all properties, events, and methods, see the [Storybook documentation](https://elevate-core-ui.inform-cloud.io/?path=/docs/components-forms-input--docs).*

## Accessibility

### Essential Accessibility Practices

**Labels are required:**
```html
<!-- Good: Always include labels -->
<elvt-input type="text" label="Full Name"></elvt-input>

<!-- Bad: Missing label -->
<elvt-input type="text" placeholder="Full Name"></elvt-input>
```

**Associate help text properly:**
```html
<!-- Good: Help text is automatically associated -->
<elvt-input 
  type="password" 
  label="Password"
  help-text="Must be at least 8 characters">
</elvt-input>
```

**Use appropriate input types:**
- Use `type="email"` for email addresses
- Use `type="tel"` for phone numbers  
- Use `type="password"` for passwords
- Use `type="url"` for website addresses

**Keyboard accessibility:**
- Tab navigates between inputs
- Enter submits forms or moves focus
- All states are keyboard accessible

**Screen reader support:**
- Labels are announced automatically
- Help text is associated with the input
- Validation states are communicated clearly
