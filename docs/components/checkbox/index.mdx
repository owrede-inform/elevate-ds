# Checkbox

Checkboxes are form controls that allow users to select one or multiple options from a list. The ELEVATE Checkbox component helps designers and developers create consistent, accessible selection interfaces that clearly communicate state and provide inclusive interaction patterns for all users.

## Overview

### When to Use Checkboxes
- **Multiple selections**: When users can choose several options from a list
- **Optional settings**: Features that can be enabled or disabled independently
- **Agreement confirmations**: Terms, conditions, privacy policies, consent forms
- **Filter controls**: Narrowing down search results or content views
- **Bulk actions**: Selecting multiple items for operations like delete or export
- **Preference settings**: User customization options that can be combined

### When NOT to Use Checkboxes
- **Single selection**: Use Radio buttons for exclusive choices from a group
- **Binary toggles**: Use Switch component for on/off states with immediate effect
- **Primary actions**: Use Button component for main interactions and submissions
- **Navigation choices**: Use appropriate navigation components for page transitions
- **Complex selections**: Consider custom selection components for advanced scenarios

## Design Principles

Understanding these core principles will help you create checkbox interfaces that reduce user errors and improve form completion rates.

### Clear State Communication Principle
**Visual design should instantly communicate the current state.** Users should never have to guess whether a checkbox is checked, unchecked, or in an indeterminate state. Ambiguous visual states lead to form errors and user frustration.

### Semantic Consistency Principle  
**Similar checkbox behaviors should look and work similarly.** Consistent interaction patterns across your application help users build muscle memory and confidence in checkbox interactions.

### Cognitive Load Reduction Principle
**Checkbox labels and groupings should minimize mental effort.** Clear, descriptive labels and logical organization help users quickly understand their options without re-reading or analyzing content.

### Inclusive Interaction Principle
**Checkboxes must work for all interaction methods.** Mouse, keyboard, touch, and assistive technology users should have equivalent experiences with clear feedback and proper focus management.

## Best Practices

### Writing Clear, Actionable Labels

**Why this matters:** Vague or confusing checkbox labels force users to guess what they're agreeing to or selecting. Clear labels reduce form abandonment and prevent user frustration with unexpected results.

:::tip Do: Use clear, specific labels that describe exactly what users are selecting
Write labels that clearly explain what happens when the checkbox is selected.

```html
<elvt-checkbox name="newsletter" value="subscribe">
  Email me about new features and updates
</elvt-checkbox>

<elvt-checkbox name="session" value="remember">
  Keep me signed in for 30 days
</elvt-checkbox>
```
:::

:::danger Don't: Use vague or confusing labels that require interpretation
Avoid generic labels that don't help users understand what they're selecting.

```html
<!-- Avoid this -->
<elvt-checkbox name="marketing">
  Communications
</elvt-checkbox>

<elvt-checkbox name="persistence">
  Remember settings
</elvt-checkbox>
```
:::

### Organizing Related Checkboxes

**Why this matters:** Poor checkbox organization increases cognitive load and makes it difficult for users to understand their options. Logical grouping and clear hierarchy help users make informed decisions efficiently.

:::tip Do: Group related options with clear headings and logical organization
Use fieldsets and legends to group related checkboxes semantically.

```html
<fieldset>
  <legend>Notification Preferences</legend>
  <elvt-checkbox name="notifications" value="email">
    Email notifications
  </elvt-checkbox>
  <elvt-checkbox name="notifications" value="sms">
    SMS notifications
  </elvt-checkbox>
  <elvt-checkbox name="notifications" value="push">
    Push notifications
  </elvt-checkbox>
</fieldset>
```
:::

:::danger Don't: Mix unrelated options without clear organization
Avoid grouping unrelated checkboxes together without semantic structure.

```html
<!-- Avoid this -->
<div>
  <elvt-checkbox name="notifications" value="email">Email notifications</elvt-checkbox>
  <elvt-checkbox name="theme" value="dark">Dark mode</elvt-checkbox>
  <elvt-checkbox name="terms" value="accepted">Terms and conditions</elvt-checkbox>
</div>
```
:::

```html

  label="Preferred contact methods"
  name="contact">
  <elvt-checkbox value="email">Email</elvt-checkbox>
  <elvt-checkbox value="phone">Phone</elvt-checkbox>

```

### Key Properties

| Property | Type | Description |
|----------|------|-------------|
| `checked` | Boolean | Whether the checkbox is checked |
| `indeterminate` | Boolean | Whether checkbox shows mixed selection state |
| `disabled` | Boolean | Whether the checkbox is disabled |
| `required` | Boolean | Whether the checkbox is required for form submission |
| `name` | String | Form field name for data submission |
| `value` | String | Form field value when checked |
| `helper-text` | String | Additional descriptive text displayed below label |
| `error-message` | String | Error message to display for validation failures |

*For complete API reference including all properties, events, and methods, see the [Storybook documentation](https://elevate-core-ui.inform-cloud.io/?path=/docs/components-checkbox--docs).*

## Accessibility

### Essential Accessibility Practices

**Proper labeling is critical:**
```html
<!-- Good: Proper label association -->
<elvt-checkbox name="terms" value="accepted" required>
  I agree to the terms and conditions
</elvt-checkbox>

<!-- Bad: Missing or unclear labels -->
<elvt-checkbox name="terms" value="accepted"></elvt-checkbox>
```

**Keyboard navigation works automatically:**
- Space toggles checkbox state
- Tab moves focus to the checkbox
- Shift + Tab moves focus away from checkbox
- Arrow keys navigate between checkboxes in groups

**Screen reader support:**
- Checkboxes announce their current state (checked/unchecked/mixed)
- Labels are properly associated and announced
- Required fields are identified to assistive technology
- Error messages are announced when validation fails

**Focus management:**
- Clear focus indicators that meet WCAG contrast requirements
- Focus remains on checkbox during state changes
- Focus moves logically through checkbox groups

**Group semantics when needed:**
- Use proper `fieldset` and `legend` for checkbox groups
- Related checkboxes are grouped semantically
- Group labels provide context for the set of options
